# SYSTEM CONTEXT — Restaurant SaaS Platform

## Goal

Build a multi-tenant, modular restaurant SaaS platform where:

- Restaurants/cafes are tenants.
- A tenant can have one or more branches.
- Each tenant gets an admin user with limited platform-level rights.
- Tenant admins can create sub-users.
- Products are the central core of the system.
- Modules such as QR Menu, Table Ordering, Home Delivery, POS, Waiter Terminal and Reporting use the same product core.
- Features are enabled/disabled based on subscription plan/entitlements.
- System supports multi-branch, multi-language, channel-based pricing, dynamic menu design and future POS synchronization.

The first phase includes:
- QR Menu
- Table Ordering / Order from Table

Future phases may include:
- POS system
- Waiter handheld terminal
- Home delivery ordering
- Advanced reporting
- Inventory
- Campaigns
- Loyalty
- Reservations

---

## Core Architectural Principle

Product is the central source of truth.

Do NOT create completely separate product identities for each module such as:

- Product
- MenuProduct
- PosSystemProduct
- OrderFromHomeProduct
- OrderFromTableProduct

Instead:

- Product = core identity
- MenuItem / POS mapping / Table order settings / Home delivery settings = module-specific projections/configurations of the same product

Correct mental model:

```text
Product Core
   ├── QR Menu projection
   ├── Table Order projection/settings
   ├── Home Delivery projection/settings
   ├── POS mapping
   └── Reporting / Order snapshot usage
```

Product should answer:

> “What is this item in our system?”

Modules should answer:

> “How is this product displayed, priced, enabled, mapped or used in this specific context?”

---

## Recommended Architecture

Start with a modular monolith.

Reason:
- Faster development
- Simpler deployment
- Easier transaction management
- Still possible to split into services later

Suggested backend module structure:

```text
/modules
  /core
  /identity
  /tenant
  /branch
  /product
  /subscription
  /qr-menu
  /table-order
  /pos-integration
  /reporting
  /import
  /pricing
  /design
```

Do not start with microservices unless scale forces it.

## Module Concept

The word “module” must be separated into levels:

```text
1. Core Platform
2. Business Modules
3. Features / Entitlements
4. Subscription Plans
```

### 1. Core Platform

These are not optional customer-facing modules. They are required for the system to work.

```text
Core Platform
- Tenant / restaurant owner management
- Branch management
- User management
- Role / permission system
- Product pool / product core
- Category core
- Media / image management
- Audit logs
- Subscription / plan / feature entitlement system
```

### 2. Business Modules

These are sellable business capabilities:

```text
QR_MENU
TABLE_ORDER
HOME_DELIVERY
POS
WAITER_TERMINAL
REPORTING
INVENTORY
CAMPAIGN
LOYALTY
RESERVATION
```

### 3. Features / Entitlements

Features are plan-controlled capabilities inside modules.

Example QR Menu features:

```text
QR_MENU_LIST
QR_MENU_CREATE_MANUAL
QR_MENU_CREATE_EXCEL
QR_MENU_CREATE_AI
QR_MENU_UPDATE_MANUAL
QR_MENU_UPDATE_EXCEL
```

Important:

> Not every feature needs its own table. Most features are access/entitlement checks.

## Product Core

### products

```sql
products(
  id,
  tenant_id,
  sku,
  base_name,
  category_id,
  tax_rate,
  status,
  created_at,
  updated_at
)
```

Purpose:
- Represents the core identity of the product.
- Does not contain module-specific fields.
- Does not contain POS name, QR menu display name, table order visibility, home delivery price, etc.

Example:

```text
Product ID: 42
base_name: Makarna
```

This means:

> “This tenant has a product called Makarna.”

It does not mean:
- How it appears in the QR menu
- How it is named in POS
- Whether it is enabled for table ordering
- Whether home delivery price is different

## Key Design Rules

### Rule 1

Do not create one giant products table with every module-specific field.

Bad example:

```text
products
- menu_price
- pos_name
- table_order_enabled
- home_delivery_fee
- qr_visible
- waiter_terminal_visible
- pos_sync_status
```

This will become unmaintainable.

### Rule 2

Do not create completely separate product identities per module.

Bad example:

```text
menu_products
pos_products
home_order_products
table_order_products
```

If all of these represent independent products, synchronization, reporting and inventory become difficult.

### Rule 3

Correct model:

```text
Product = identity
Module-specific tables = configuration/projection/mapping
OrderItem = snapshot
```

### Rule 4

Feature availability should be controlled by entitlement checks, not hardcoded logic.

Example:

```text
Can tenant use dynamic pricing?
Check feature QR_MENU_DYNAMIC_PRICING.
```

### Rule 5

Menus and design are presentation layers.
Products and pricing are domain/core layers.

## Important Open Questions

These decisions should be clarified before implementation:

1. Will one product be shared across all branches by default?
2. Can branches override product name, price, visibility and availability?
3. Should a menu belong to one branch or multiple branches?
4. Should categories be global product categories or menu-specific categories?
5. Should POS sync be one-way or two-way?
6. Should product deletion be soft-delete only?
7. Should price changes be versioned?
8. Should menu design rules be evaluated on server-side or client-side?
9. Should QR menu work without login/publicly?
10. Should table ordering require staff approval before sending to kitchen/POS?
11. Will there be stock/inventory in future?
12. Will products have variants/options such as size, sauce, extra cheese?
13. How will tax/VAT be handled per product/channel?
14. Will campaign pricing and dynamic pricing conflict? If yes, which has priority?
15. Will reporting be real-time or periodically aggregated?

## Final Summary

Use this core principle:

```text
Product Core is the stable identity.
Modules are business capabilities.
Features are plan-controlled abilities.
Module-specific product tables are projections/settings/mappings.
Orders store snapshots.
Pricing and design should be rule-based and reusable.
```
